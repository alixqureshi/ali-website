What a loop actually is

Same tool, two ways to use it. One keeps you in the chair as the quality check. The other takes you out of it.

Prompting

You ask. It answers. Done.

  • Stops the second it replies — good or not
  • You catch what's wrong, ask again
  • You're the QA. Every time.
Looping

You set a goal. It runs until it hits it.

  • Checks its own work against the goal
  • Not there? Rewrites and tries again
  • Repeats on its own until it passes

The unlock isn't a better prompt. It's giving the AI a finish line it can check itself against — and the permission to keep going until it gets there.

The 3 ingredients of a loop

Miss any one of these and it's just prompting with extra words. Get all three and it runs itself.

  1. A goal it can grade itself against. A clear pass/fail — a test, a checklist, "the page loads with no errors," "every claim has 3+ links that open." Something it can check without asking you. This is the make-or-break one.
  2. A way to see the result. It has to actually run the code, open the page, read the error, or step through the checklist. Real feedback, not a guess. Claude Code already does this out of the box.
  3. Permission to keep going until it passes. Say it out loud: don't stop at the first attempt. Build → test → read the failure → rewrite → re-test, until the bar is met.

The one rule that makes or breaks it

This is the part most people get wrong. "Make it good" can't be graded — there's nothing to check against, so the loop stops at the first answer. Give it a number, a checklist, or a clear pass/fail and it can self-grade.

"Make it good."
"Don't stop until the page loads with zero errors."
"Write a strong landing page."
"Write it for our exact ICP. If it misses, rewrite it."
"Make the research solid."
"Every claim needs 3+ links. Open each to confirm it backs the claim."
"Punch up the copy."
"It has to pass all 7 of my hook checks before it's done."
"Build it well."
"Build it, run it, and don't stop until it works."

The test: could a stranger look at the output and tell you pass or fail without guessing what you meant? If yes, the AI can too.

Turn any prompt into a loop

You don't need code. In Claude Code, you turn a normal request into a loop by bolting on two things: a goal it can check, and "don't stop until it passes."

Prompting

"Build me a landing page with this logic."

  • Builds it, hands it over, stops. You find the bug. You ask again.
Looping

"Build it. Then run it and open it. If anything errors, fix it and try again. Don't stop until it loads clean and does X, Y, Z."

  • Builds → tests → reads the failure → rewrites → re-tests, until your check passes.

The formula: the task + a check it can verify itself + "don't stop until it passes." That's a loop.

5 loops you can steal

Real loops — the kind I run building the AI systems behind a $3M agency. Swap the brackets for your specifics and paste them straight into Claude Code.

1. Build a page until it works

paste into Claude Code
Build [the page] with this logic: [paste logic]. Then run it and open it in the browser. If anything errors or the logic doesn't behave, fix it and try again. Don't stop until it loads clean and every step of the logic works.

Why it loops: "it works" either is or isn't true — the AI can run it and see.

2. Write copy for a specific ICP

paste into Claude Code or a Claude Project
Write this landing page for our exact ICP: [who they are, what they want, what they fear]. Then read it back as that person. If a single line wouldn't land for them, rewrite it. Don't stop until every line speaks to that ICP.

Why it loops: "made for this ICP" is a bar it can re-read against, line by line.

3. Research with real sources only

the anti-hallucination loop
Write a one-page brief on [topic]. Every claim needs 3+ links. Open each link and confirm the source is real and actually backs the claim. Drop any link that doesn't. Don't stop until every claim has 3 working sources that hold up.

Why it loops: it goes link by link instead of trusting itself — exactly where AI usually hallucinates.

4. Hold copy to your own checklist

turn your standards into a grader
Here's my checklist for a good [hook / script / email]: [paste your checklist]. Write the [asset], then score it against all of them. For any it fails, rewrite and re-score. Don't stop until it passes every item on the list.

Why it loops: your checklist is the grader. This is how you keep a quality bar without checking every output yourself.

5. Fix until the tests pass

for anything with a test suite
Fix [the bug / feature]. After each change, run the tests. If any fail, read the output, fix the cause, and run them again. Don't stop until the full suite is green.

Why it loops: the test suite is the most honest grader there is — green or not.

When you want it in code

For most of what you'll do, Claude Code is the loop — you just give it the finish line and the means to check. Written literally, the same idea is a while loop:

the loop, in plain pseudocode
while goal not met:
    ask the AI to act (with tools)
    let it run the code / open the page / read the result
    check the result against your goal
    if it passes  -> stop
    else          -> feed the result back and go again

Call the model, let it act, check the stop condition, repeat until met. The Claude Agent SDK is built for exactly this — but you don't need a line of code to start. Give Claude Code a gradeable goal and "don't stop until it passes," and you're already looping.

What kills a loop

  • A vague goal. "Make it good" gives it nothing to check. Number, checklist, or pass/fail — always.
  • No way to see results. If it can't run it, open it, or read the output, it's guessing.
  • Forgetting "don't stop." Without it, it stops at attempt one. Say it explicitly.
  • Stepping in mid-run. Let it fail and self-correct. The moment you take over, you're the machine again.

The whole thing in one line: you stop steering every step and start setting the finish line. Set the bar once, and it holds without you in the room.
